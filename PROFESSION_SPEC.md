## 1. What Is a Professional Operating System?

A **Professional Operating System (OS)** is a structured collection of plain markdown files that encodes how a professional thinks, decides, and operates in their role. Uploaded to an AI platform (ChatGPT, Gemini, Claude), these files give the AI the full context of the person's philosophy, their team or client relationships, their organisation's constraints, and their current quarter — enabling structured, context-aware decision support.

The OS metaphor is intentional: like a computer's operating system, a Professional OS has:
- A **kernel** — the portable, permanent core (values, decision logic)
- **Drivers** — swappable context files (current team, current clients)
- **Processes** — ephemeral execution state (current quarter's plan)
- A **command interface** — named operations the AI copilot executes on demand

The framework has three properties that make it useful in practice:
1. **Portable** — the logic (how you decide) travels with you across roles, companies, and client cohorts. The data (who you work with right now) stays behind.
2. **AI-native** — files are written as AI-readable documents, not for humans to scan. Formatting, section numbers, and canonical source markers are designed for AI retrieval, not for human aesthetics.
3. **Decision-support, not decision-making** — the OS calculates trade-offs and flags issues; humans make the final call. Rule Zero (§6) enforces this unconditionally.

## 2. The Three Content Tiers

Every file in a Professional OS contains content from one or more of these tiers. Understanding the tiers is essential for building and validating any profession variant.

| Tier | Who Provides It | When Created | Example Content |
|------|----------------|--------------|----------------|
| **Tier 1: Structure** | This specification (framework) | Once — same for all professions | Layer layout (00–06), file header format, command format, freshness system, portability contract, Rule Zero |
| **Tier 2: Profession** | `GENERATOR_PROMPT.md` output, or hand-crafted by a domain expert | Once per profession variant | Kernel philosophy sections, decision gate names, capacity categories, leadership/coaching modes, boardroom personas, command names, Setup Wizard questions |
| **Tier 3: Personal** | `SETUP_WIZARD.md` output — always the end user | Once per user, refreshed quarterly | `personal_dna.md` content, company/practice config, team/client roster, entity cards, quarterly plan |

**Why this matters:**
- A pre-built profession (e.g., "Manager OS") covers Tier 1 + Tier 2. It is *ready to personalise*, not *ready to use*. The user must run the Setup Wizard to fill Tier 3.
- Pre-built professions should include realistic **example content** for Tier 3 files so users can see what "good" looks like. The Setup Wizard replaces example content with the user's real data. Mark all example content with: `<!-- EXAMPLE CONTENT: Replace with your actual data using the Setup Wizard -->`
- The `GENERATOR_PROMPT.md` creates Tiers 1 + 2 + a profession-specific Setup Wizard. It does NOT create Tier 3 — that is always personal.

## 3. The 7-Layer Model

> **MUST** — Every Professional OS must have exactly these seven layers with these folder names. No profession may add, remove, or rename layers.

| Layer | Folder | Lifecycle | Portable | Purpose |
|-------|--------|-----------|----------|---------|
| 00 | `00_BOOT/` | Permanent | Yes | System manual and bootstrap prompt. Documents the architecture and tells the AI how to load the OS. Read once. |
| 01 | `01_KERNEL/` | Permanent | Yes | The professional's portable philosophy — values, decision protocol, personal DNA. Travels with the person across every context. |
| 02 | `02_CONFIG/` | Per-context | No | External context — the company, practice, or organisation the person currently operates within. Replaced when context changes. |
| 03 | `03_DRIVERS/` | Per-roster | No | Current people — team members, clients, stakeholders, or accounts the person currently serves or leads. Replaced when roster changes. |
| 04 | `04_PROCESSES/` | Per-quarter | No | Current execution — the tactical or quarterly plan. Replaced every quarter. |
| 05 | `05_COMMANDS/` | Permanent | Yes | The AI interface — all named command definitions and the system prompt. Travels with the person. |
| 06 | `06_BOARDROOM/` | Permanent | Yes | A virtual advisory council — expert persona definitions for high-stakes decisions. Travels with the person. |

### 3.1 Required Files Per Layer

> **MUST** — Each layer must contain at minimum the files listed. Additional files are allowed.

| Layer | Required Files | Naming Convention |
|-------|---------------|-------------------|
| `00_BOOT/` | `README.md`, `bootstrap_prompt.md` | Fixed names |
| `01_KERNEL/` | `{profession}_operating_system.md`, `{profession}_decision_protocol.md`, `personal_dna.md` | `{profession}` = file prefix (e.g., `manager`, `coach`, `product_manager`) |
| `02_CONFIG/` | `{context}_operating_system.md`, `{context}_strategy.md` | `{context}` = operating context (e.g., `company`, `practice`, `agency`) |
| `03_DRIVERS/` | `{roster}_operating_system.md` OR `{roster}_portfolio.md`, `{entity}_card - [Name].md` (one per person/client) | `{roster}` = roster name; `{entity}` = individual type |
| `04_PROCESSES/` | `{plan_type}_plan.md` | `{plan_type}` = plan name (e.g., `tactical`, `quarterly`, `sprint`) |
| `05_COMMANDS/` | `command_reference.md`, `system_prompt.md` | Fixed names |
| `06_BOARDROOM/` | `boardroom.md` | Fixed name |

## 4. The Portability Contract

> **MUST** — Every Professional OS must define and honour this contract. When the user changes context, the layers behave as follows:

| Action | Layers Affected | Description |
|--------|----------------|-------------|
| **KEEP** (travels with you) | `01_KERNEL/`, `05_COMMANDS/`, `06_BOARDROOM/` | Your philosophy, decision logic, and advisory council are yours — they survive any company or team change |
| **UPDATE** (adapt to new context) | `02_CONFIG/` | Your new employer's mission, strategy, and values replace the old ones |
| **REPLACE** (start fresh) | `03_DRIVERS/`, `04_PROCESSES/` | New team members / clients and a new quarterly plan from scratch |

The portability contract exists to separate **logic** (how you decide — portable) from **data** (who you work with right now — replaceable). It makes the OS reusable across contexts.

## 5. File Header Standard

> **MUST** — Every file in every layer must begin with this exact header format. Machine-parseable. No deviations.

```markdown
# [File Title]

> **Layer:** [LAYER_NAME]
> **Owner:** [Owner Name]
> **Version:** [YYYY.MM]
> **Last Updated:** [YYYY-MM-DD]
> **Portable:** [Yes — reason] / [No — reason]
```

**Field definitions:**

| Field | Who Updates | When | Purpose |
|-------|------------|------|---------|
| `Version` | Framework maintainers | Per release (e.g., monthly) | Tracks which version of the profession template this file implements |
| `Last Updated` | The user (end professional) | When running `prep_refresh` or `quarterly_reset` | Tracks when file content was last refreshed — drives freshness warnings |
| `Portable` | Set at template creation | When the file is created | Tells the AI copilot whether this file travels with the user across contexts |

## 6. Rule Zero — Universal Safety Mechanism

> **MUST** — Every Professional OS must implement Rule Zero **verbatim** in the Decision Protocol. The text is identical across all professions; it cannot be modified or removed.

```markdown
### §0 — Rule Zero

The OS is a decision-SUPPORT tool. It does not make decisions.

When OS rules produce any of the following:
- Conflicting outputs from two or more gates
- A recommendation that feels wrong in context
- Ambiguity that cannot be resolved by re-reading the relevant files
- A "bad gut feeling" that persists after analysis

**Stop. Invoke Rule Zero. Have a synchronous human conversation.**

Rule Zero supersedes all other rules. Any human can invoke it at any time, including
the AI copilot if it detects unresolvable ambiguity. It cannot be overridden by
any other rule in the OS.

The action is always the same: talk to a human, directly, before deciding.
```

## 7. Decision Protocol Template

> **MUST** — Every Professional OS must implement §0 (Rule Zero, verbatim) and §1–§4 below.
> **SHOULD** — Three decision gates is the recommended default. Professions with four distinct constraint levels may use four. Document the deviation in the OS `00_BOOT/README.md`.

The Decision Protocol lives at `01_KERNEL/{profession}_decision_protocol.md`. It has five sections:

### §0 — Rule Zero
Copy verbatim from §6 of this specification.

### §1 — Hierarchy of Precedence (Decision Gates)

Define 3 gates (default) ordered from most-local constraint to most-global. The gate that most often blocks execution should be Gate 1. Example structures:

| Gate | Manager OS | Coach OS | Pattern |
|------|-----------|---------|---------|
| Gate 1 | Team Capacity | Client Fit | Local constraint — what blocks at the ground level |
| Gate 2 | Company Strategy | Practice Strategy | Strategic alignment — North Star check |
| Gate 3 | Individual Match | Capacity | Flexible boundary that can stretch if Gates 1–2 pass |

For each gate, define:
- Gate name and what it checks
- Pass condition (what allows moving to the next gate)
- Fail condition (what triggers reject or negotiate)
- Source of truth (which file/section to read)

### §2 — Interaction Rules

Define three functions:
- **Override:** When a higher-priority rule overrides a lower-priority one (provide a concrete example from the profession)
- **Conflict:** What happens when two gates produce contradictory outputs (return to Rule Zero or escalation path)
- **Exception:** A crisis scenario that suspends normal gate sequencing (define the exception conditions and recovery protocol)

### §3 — New Request Loop

Define the iterative protocol for processing new requests:
- Initial evaluation sequence (gate 1 → gate 2 → gate 3 or Rule Zero)
- Re-entry rules (when a rejected request can re-enter the gates)
- Maximum iteration rule (how many re-entry cycles before hard rejection)
- Rule Zero override trigger (the condition that exits the loop entirely)

### §4 — Decision Speed (Reversibility Framework)

Define the reversibility classification:
- **Fast decisions (reversible):** Define what "reversible" means for this profession. Examples from profession context.
- **Slow decisions (irreversible):** Define what "irreversible" means for this profession. Examples. Require consultation before acting.
- Provide 5–7 concrete examples of common decisions classified as fast or slow.
- Define Pressure Mode interaction: which decisions become slower under Pressure Mode?

## 8. Command System

### 8.1 Command Format

> **MUST** — Every command in `command_reference.md` must follow this exact structure. No field may be omitted.

```markdown
### `command_name [optional_args]`

**Trigger:** [When to run this command — describe the situation]
**Purpose:** [What structured output the command produces]

**OS Files to Read:**
- `[file path]` — [specific sections to reference]

**Ask User For:**
- [Live data that must come from the user — never assumed or invented]

**Critical Files for Freshness:**
- [emoji] `[file]` — [grace period] ([reason this file must be current])

**Output Format:**
[Exact template in markdown that the AI must produce — including section headers,
tables, and any emoji status indicators]

**Chains With:** `[other_command]` — [why this command's output feeds the next]
```

**Freshness emojis:** Use `🔴` (0-day — must be current), `🟡` (3–7 day — warn if stale), `🟢` (30-day — reference only).

### 8.2 Required Command Categories

> **MUST** — Every Professional OS must include commands in all five categories with at minimum the commands listed.

| Category | Required Minimum Commands | Purpose |
|----------|--------------------------|---------|
| **Lifecycle** | `init_week`, `init_day`, `shutdown_day`, `shutdown_week` | Daily/weekly bookends — planning at start of period, review at end |
| **People** | At least one `prep_{interaction} [name]`, at least one `onboard_{entity} [name]` | Prepare for interactions with team/clients/stakeholders; structured onboarding |
| **Execution** | At least `prep_{planning_type}`, `capacity_check`, `retro` | Planning cycle management, capacity monitoring, retrospectives |
| **Decision Support** | At least one `{request_type}_request [desc]`, one `{boundary_type}_check [topic]`, `quarterly_reset`, `boardroom [topic]` | Structured decision-making, boundary management, periodic review |
| **Maintenance** | `prep_refresh`, `version_upgrade [version]` | System health monitoring, OS version upgrades |

Additional commands beyond the minimum are welcome. They must follow the §8.1 format.

## 9. Boardroom System

> **MUST** — Every Professional OS must include a Boardroom (`06_BOARDROOM/boardroom.md`) structured as follows.
> **SHOULD** — 6 personas is the recommended number. The acceptable range is 4–8. Document the deviation in `00_BOOT/README.md` if outside the default.

### 9.1 Persona Definition Template

Each persona must include:

```markdown
### [Name] — [Archetype Label]

> "[Worldview quote — one sentence defining this persona's lens]"

**Focus Areas:**
- [Area 1]
- [Area 2]
- [Area 3]

**Probing Questions:**
1. [Question this persona would always ask — specific to their worldview]
2. [Second probing question]
```

### 9.2 Required Boardroom Sections

1. **§1 — Persona Roster:** All personas listed with the template above. Include an archetype ↔ persona reference table mapping the 6 archetype slots to their named personas.
2. **§2 — Working Principles:** Minimum 4 principles that govern all boardroom sessions. Must include: "Focus on the user's personal DNA and constraints," "Challenge at least one assumption per session," "Be concrete — actionable over theoretical."
3. **§3 — Activation Logic:** A table mapping topic types to the personas that activate for each topic type. This tells the AI which 2–4 personas to voice when a specific topic is raised.
4. **§4 — Session Format:** The standard output structure for a `boardroom [topic]` call — how personas voice perspectives, how synthesis is produced, where Rule Zero reminder appears.
5. **§5 — The [Profession Role]'s Chair:** The user's current context block — their active goals, current capacity, and declared constraints. Updated during `prep_refresh`.

### 9.3 The 6 Archetype Slots

> **SHOULD** — These archetypes represent the recommended balance of perspectives. Map each to a domain-appropriate figure for the profession.

| Slot | Archetype | Function | Manager OS Example | Coach OS Example |
|------|-----------|----------|-------------------|-----------------|
| 1 | The Action Driver | Pushes execution, cuts analysis paralysis | Andy Grove | Tony Robbins |
| 2 | The Systems/Revenue Thinker | Optimises structure, unit economics, leverage | Gene Kim | Alex Hormozi |
| 3 | The People/Purpose Voice | Protects the human side, long-term meaning | Bill Campbell | Simon Sinek |
| 4 | The Truth Teller | Forces honesty, names the uncomfortable thing | Kim Scott | Brené Brown |
| 5 | The Standards Enforcer | Holds the professional bar, challenges fragility | Nassim Taleb | Marshall Goldsmith |
| 6 | The Grounding Anchor | Keeps the user centred, prevents burnout spiral | Brené Brown | Richard Bolles |

## 10. File Freshness System

> **MUST** — Every Professional OS must implement file freshness validation in every command.

### 10.1 Grace Period Tiers

| Symbol | Grace Period | Meaning | Typical Use |
|--------|-------------|---------|-------------|
| 🔴 | 0 days | Must be current before command runs. If stale, block and ask user to update. | `quarterly_plan.md` in `init_week` |
| 🟡 | 3–7 days | Warn if stale, but proceed. Output a `📋 File Status` block before command output. | `client_portfolio.md` in `prep_session` |
| 🟢 | 30 days | Reference only. Note if stale, do not warn. | `coach_operating_system.md` in most commands |

### 10.2 Validation Logic

Every command that declares `Critical Files for Freshness` must apply this logic before producing its main output:

```
freshness_check(files):
  stale_files = []
  for each file in files:
    days_since_update = today - file.last_updated
    if days_since_update > grace_period:
      stale_files.append(file, grace_period, days_since_update)

  if any(file.grace_period == 0_day for file in stale_files):
    output "📋 File Status — BLOCKED"
    list stale 0-day files
    output "Run prep_refresh to update before continuing."
    STOP

  if any(file.grace_period == 3_7_day for file in stale_files):
    output "📋 File Status — WARNINGS"
    list stale 3-7 day files
    output "Proceeding with caution. Run prep_refresh when possible."

  continue with command output
```

## 12. Extension System: Skills

> **SHOULD** — Skills are optional capability profiles that augment how the AI copilot performs tasks. They are NOT a layer of the 7-layer model. The 7-layer model is unchanged.

### What Is a Skill?

A **skill** is a passive AI capability profile — a structured document that tells the AI *how* to perform a class of task (e.g., write in the user's voice, take notes in the user's format). Skills differ from commands:

| | Commands (`05_COMMANDS/`) | Skills (`skills/`) |
|-|--------------------------|-------------------|
| **Triggered by** | User typing a command | AI detecting a relevant task |
| **Output** | Structured decision-support result | Enhanced quality of any output |
| **Lifecycle** | Permanent (part of the OS) | Permanent, portable |
| **Required?** | Yes (minimum 5 categories) | No — entirely optional |

### The `skills/` Directory

Skills live in a `skills/` directory at the **reference root** — a sibling to `00_BOOT/` through `06_BOARDROOM/`, **outside** the 7-layer model. This means:

- Skills do NOT participate in the layer lifecycle (portability contract, freshness system, bundle layer numbering)
- Skills ARE portable — they travel with the professional like the Kernel does
- The `skills/` folder may be empty or absent — this is valid and expected for new OS users

```
references/coach-os/
├── 00_BOOT/ … 06_BOARDROOM/   ← The 7-layer OS (unchanged)
├── skills/                     ← Extension system (sidecar)
│   └── writing_voice.md        ← A skill file
└── bundle.conf
```

### Skill File Format

> **MUST** — Every file in `skills/` must follow this format.

```markdown
# Skill: [Skill Name]

> **Layer:** SKILLS
> **Owner:** [Your Name]
> **Version:** [YYYY.MM]
> **Last Updated:** [YYYY-MM-DD]
> **Portable:** Yes — travels with the professional across all contexts
> **Type:** Universal | Profession | Custom

## What Is This Skill?
[1–2 sentence description of what capability this skill gives the AI copilot]

## Activation
[When should the AI load and apply this skill? e.g.: "Activate when generating any written content on behalf of the user — emails, posts, proposals, documents."]

## Generation
[How this skill was created — either: "Generated via `skills-library/[skill-name]/generation_prompt.md`" or a custom description]

## Profile
[The generated content — voice analysis, style guide, output patterns. This section is populated by running the generation prompt.]
```

### Skill Types

| Type | Description | Example |
|------|-------------|---------|
| **Universal** | Applicable to any profession — not context-specific | Writing Voice |
| **Profession** | Specific to a profession variant | Career Coach Session Note Style |
| **Custom** | User-created, not from the shared skills library | Any |

### The Skills Library

The `/skills-library/` directory at the **framework root** contains universal skill templates shared across all profession variants:

```
/skills-library/
└── writing-voice/
    ├── README.md               ← What this skill enables
    └── generation_prompt.md    ← The meta-prompt for generating the skill
```

Each skill in the library provides a `generation_prompt.md` — a meta-prompt the user runs in their AI to produce their personalised skill file.

### Bundle Behaviour

When `bundle.sh` runs and a `skills/` directory exists with `.md` files, it produces a **second output file**: `bundle/skills_compiled.md`. This is uploaded to the AI platform alongside the primary OS bundle — two files total.

If `skills/` is empty or absent, `bundle.sh` skips the skills pass silently.

## 13. Professions That Fit This Model

The Professional OS framework is not a general-purpose productivity system. It is designed specifically for professions with the following characteristics.

### Required Characteristics

A profession fits this model if it has **all five**:

1. **Structured recurring cadences** — Clear daily, weekly, and quarterly rhythms (standups, 1:1s, planning cycles, retrospectives). Without cadences, Lifecycle commands have nothing to anchor.
2. **People relationships** — The professional serves, leads, or manages a defined group (team, clients, stakeholders, accounts). Without people, Drivers layer has nothing to hold.
3. **Measurable goals and capacity constraints** — There are explicit goals per quarter and a finite capacity for work. Without this, capacity_check and quarterly_reset are meaningless.
4. **Recurring high-stakes decisions** — Decisions that arise regularly, have meaningful consequences, and benefit from structured deliberation (hiring, scope cuts, client acceptance, escalations).
5. **Contextual portability** — The professional changes teams, companies, or client cohorts over time but carries a consistent philosophy and decision framework across those changes.

### Known Good Fits

| Profession | Primary People Layer | Key Cadence | Notes |
|-----------|---------------------|-------------|-------|
| Engineering Manager | Direct reports (team) | Sprint / 1:1 / OKR quarter | Reference implementation available |
| Career Coach | Clients (1:1 coaching) | Weekly sessions / programme milestones | Reference implementation available |
| Product Manager | Stakeholders + squad | Sprint / PI planning / quarterly | Strong fit — clear Gates and capacity model |
| Design Lead | Designers + partners | Critique cycles / project milestones | Good fit if leading a team, weaker if individual |
| Sales Director | Accounts + sales team | Pipeline reviews / QBRs / annual targets | Good fit — natural capacity and gate model |
| HR Business Partner | Business leaders + employees | Org cycles / performance reviews / headcount | Good fit with strong people layer |

### Known Poor Fits

| Profession | Why It Doesn't Fit |
|-----------|-------------------|
| Individual Contributor (no reports) | No people layer — Drivers is empty. The OS works but is significantly reduced. |
| Founder / CEO | Scope too broad — the OS models one function, not an entire organisation. |
| Freelance Artist (project-based) | No recurring cadences — each project is ad-hoc. Lifecycle commands have no anchor. |
| Therapist / Clinical Psychologist | Professional ethics prohibit any AI involvement in clinical decisions. Rule Zero must not be AI-assisted in this context. |

1. Navigate to the reference directory (`cd references/coach-os/`)

1. Read this specification fully
2. Use `GENERATOR_PROMPT.md` — the AI meta-wizard that interviews you and generates a complete profession variant
3. Upload both reference implementations as examples to the AI during generation
4. Validate your generated OS against this specification before considering it stable
